# ADR-005: 存储层 SQLite → PostgreSQL 渐进迁移

**状态**：采纳  
**日期**：2026-07-14  
**决策者**：sg-architect  
**来源**：自发  

---

## 上下文

MACS 需要持久化 Agent 状态、审计历史、Token 用量趋势等。需要决定存储方案。OPC 场景要求零配置（不能假设用户有 PostgreSQL），但未来多租户需要关系型数据库。

## 决策

v1 使用 SQLite（单文件、零配置），v2 迁移到 PostgreSQL（多租户、并发写入）。

## 理由

1. **OPC 优先**：目标用户是个人操作者，docker compose up 就能跑
2. **渐进复杂度**：SQLite 可以处理单 OPC 的吞吐，瓶颈在多租户时才出现
3. **迁移路径明确**：FastAPI + SQLAlchemy 的 ORM 层使存储切换成本低

## 后果

| 影响 | 描述 |
|------|------|
| 正面 | v1 零配置，docker compose up 即用 |
| 负面 | v1→v2 需数据迁移脚本 |
| 风险 | 如果 v1 快速获得多用户，可能需要提前迁移 |

## 备选方案

| 方案 | 评估 | 原因 |
|------|:----:|------|
| A: v1 直接上 PostgreSQL | ❌ | 破坏零配置原则，OPC 用户可能没有 PostgreSQL |
| B: SQLite 永久方案 | ❌ | 多租户场景下并发写入瓶颈明显 |
